# 스마트팜 에이전트 작업체계 계획 — 3단계 사상 비판적 검토

작성일 2026-07-02 · 대상: agsmartroi_260121.pptx(41p 플랫폼 비전) vs 현재 엔진(4축 결정론 코어)

## 0. 결론 (먼저)

제안된 **쪼개기 → 구분하기 → 자동화하기 → 실행**은 방향은 옳다.
그러나 **쪼개는 축을 "업무단계(Phase 1-10)"로 잡으면 근본문제를 못 푼다.**
쪼개고 구분해야 할 진짜 축은 **신뢰 유형**이다:

1. **결정론(Deterministic)** — 입력이 같으면 항상 같은 값. → 엔진이 계산. (자동화 O)
2. **시세성(Volatile)** — 단가·노임·유가처럼 시점마다 변함. → 주입만. (자동화 X, 인터페이스만)
3. **판단성(Judgment)** — 내재해 지정값·작물 레시피·전문가 해석. → 보조/초안까지. (자동화 X)

이 경계선이 곧 **자동화 가능선**이다. 이 선을 넘어 자동화하면 발표자료가 지적한 근본문제가 그대로 재발한다.

## 1. 근본문제 재정의 (덱의 고통점 → 진짜 문제)

덱 S4~S6이 말하는 현장 고통점:
- 투자비 괴리율 평균 **22%**, 난방비 예측 오차 **±30%**, 비용 원인추적 불가(15개 중 8개).
- 즉 고객의 불만은 "기능이 없다"가 아니라 **"숫자를 믿을 수 없고, 왜 그런지 추적할 수 없다"**이다.

따라서 **근본문제 = 숫자의 신뢰성·재현성·추적성 결여.**
성공 기준은 화려한 기능 수가 아니라, 모든 산출 숫자가
(a) **단일 출처(엔진)에서 재현**되고, (b) **근거가 추적**되며, (c) **추정/확인요망이 명시**되는 것.

→ 이미 채택한 원칙(엔진 단일 출처 · 시세성 주입 · 근거 명시 · `[추정]` 표기)이 **정확히 근본문제를 겨냥**하고 있다.
   반대로 덱의 야심(Digital Twin·Monte Carlo·RAG·AI대화)을 근거·추적 없이 얹으면 근본문제를 **되살린다.**

## 2. 3단계 사상의 비판적 재해석

| 단계 | 순진한 해석 (위험) | 올바른 재해석 (채택) |
|---|---|---|
| **쪼개기** | 덱의 Phase 1-10 / 5스텝으로 분할 | **검증 가능한 최소 단위(축별 순수함수)**로 분할. 각 조각은 자체 회귀테스트 보유. Phase는 표현 순서일 뿐 자동화 단위가 아님 |
| **구분하기** | 기능 카테고리 분류(설계/시공/경제성…) | **에피스테믹 분류**(결정론/시세성/판단성). 이 경계가 자동화 가능선 |
| **자동화하기** | 덱 기능 전부 자동화 | **결정론만 자동화.** 시세성은 주입 인터페이스, 판단성은 초안/설명까지만. 과자동화 = 근본문제 재발 |
| **실행** | 데모 한 번 돌려보기 | **고정 회귀 케이스(원채원)로 매번 재현 검증.** 재현이 곧 신뢰의 증거 |

핵심: 이 방법론은 이미 암묵적으로 실행돼 왔다(엔진=자동화된 결정론, FarmInput=주입, 테스트=실행검증).
명시적으로 못 박을 것은 **"쪼개기/구분하기의 축은 신뢰 유형"**이라는 점 하나다.

## 3. 컨텍스트 한계 보완 = 상태의 외부화

컨텍스트에 다 담으려 하지 말고, **durable artifact로 밀어낸다.** 각 세션은 필요한 조각만 로드.

| 외부화 대상 | 파일 | 역할 |
|---|---|---|
| 계산 로직 | `smartfarm_engine.py` | 유일 계산 출처 |
| 재현 증거 | `test_engine.py` | 회귀 고정 |
| **근거 대장(신설 필요)** | `엔진데이터_레지스트리.md/json` | 모든 상수·밴드·표의 출처·기준시점·`[추정]` 태그 |
| 케이스 저장소(신설) | `cases/*.json` | 농가별 입력·결과 스냅샷 |
| 진행 기억 | `인수인계_스마트팜에이전트.md` | 세션 간 연속성 |

**지금 최대 리스크(근본문제 관점):** 근거문서(`SmartFarm_엔진데이터.md`·지침)가 폴더에 **없다.**
벤치마크는 6건(원 8건), SPEC은 대표 12종(원 26종), 일부 값은 `[추정]`/`[확인요망]`.
→ **자동화를 넓히기 전에 provenance(근거 대장)부터 세워야** 신뢰성 근본문제를 실제로 푼다. 모래 위 자동화 금지.

## 4. 통합 작업체계 (Integration Contract)

**단 하나의 계약:** 모든 상위 기능(웹앱·정적사이트·향후 Digital Twin·AI 인터페이스)은
**엔진 출력의 소비자/렌더러일 뿐, 계산을 재구현하지 않는다.**

- 새 기능 = **새 엔진 모듈 + 자체 테스트 + 근거**. 병렬 계산기(예: 엑셀 수식, JS 계산) 금지.
- 데이터 스키마 3종으로 통합:
  - `FarmInput` — 시세성·현장 입력(주입)
  - 엔진데이터 레지스트리 — 결정론 상수의 근거
  - 케이스 레지스트리(JSON) — 입력·결과 스냅샷(비교·저장·재현)
- AI/대화 계층은 **엔진 숫자를 "설명"만** 한다. 숫자 생성·창작 금지, 근거 인용은 레지스트리에서.

## 5. 우선순위 로드맵

| 우선 | 작업 | 근본문제 기여 | 선행조건 |
|---|---|---|---|
| **P0** | **근거 대장(provenance registry)** — 엔진 상수·밴드·SPEC·환경가중치에 출처·기준시점·`[추정]` 태깅. 근거문서 확보 요청 | 신뢰성·추적성의 토대 | 근거문서(있으면 반영, 없으면 "현재값·미검증" 태깅) |
| **P1** | **경로 C 마감** — 춘천 리포트 index 연결 · 케이스 저장/목록(JSON 레지스트리) · 케이스 비교 뷰 | 재현·비교 가능성 | P0 스키마 위에 얹으면 최선(없이도 가능) |
| **P2** | 확장(근거 있을 때만) — OPEX 15항목 분해 · 민감도/what-if 시나리오 | 원인추적·의사결정 | 엔진 모듈 + 테스트 |
| **P3** | AI/판단 계층 — 대화형 원인설명(엔진 숫자 해석·RAG 근거인용). 계산 금지 | 접근성 | P0~P2 산출물 |

**비판적 권고:** 요청된 "경로 C 마감(P1)"은 가치 있으나, 그 위에 쌓을 신뢰의 토대(P0)가 비어 있다.
케이스 저장/비교를 만들 때 **케이스 스키마에 근거·기준시점 필드를 함께 넣어** P0의 일부를 자연스럽게 흡수하는 것이 최선(중복작업 방지).

1. **근거문서 확보 가능?** (`SmartFarm_엔진데이터.md`, 지침, 원본 엑셀) — 있으면 P0가 검증형, 없으면 "현재값+미검증" 태깅형으로 진행.
