## 1. 한 문단 요약
스마트팜을 **진단→설계→시공→경제성** 4축으로 자동 산정하는 에이전트를 만들고 있다. 4축의 계산 로직은 이미 순수 함수로 `smartfarm_engine.py`에 고정되어 있고, 회귀 테스트 11개를 모두 통과한다. GPT 없이도 계산이 재현되도록 설계된 **결정론적 코어**가 완성된 상태이며, 다음 단계는 이 엔진 위에 사용자용 산출물(엑셀 계산기 또는 웹 리포트)을 얹는 것이다.

## 3. 엔진이 하는 일 (함수 지도)
`smartfarm_engine.py` — 축별 핵심 함수:

- **설계**: `select_specs` (지역 적설심·풍속 → 내재해형 규격 필터, 형식별 최소사양)
- **난방**: `heating_load` (최대난방부하·난방기용량·연료소비), `verify_heating_vs_actual` (실측 이중검증, 자릿수 오류 감지용)
- **시공-골조**: `greenhouse_total_estimate`(방법 A: 온실 전체 개산) / `structure_only_estimate`(방법 B: 골조 단독) — **둘을 절대 혼용하지 말 것(이중계산 방지).**
- **시공-검증**: `benchmark_check` (총액÷면적 → 피복별 밴드 대조: 필름 115~220 / 유리 180~230 / 불소 230~310 천원/㎡)
- **시공-제경비**: `apply_overheads` (보험·안전·일반관리·이윤 → VAT 전 총액)
- **경제성**: `env_fitness`·`yield_adjustment`·`production_kg`(환경적합도→생산량), `finance`(ROI·Payback·NPV·IRR·보조금 반영 실질ROI), `improvement_roi`(리뉴얼), `cluster_economics`(단지)

**설계 원칙 4가지 (반드시 유지):**
1. 데이터를 열어 확인하고, 실행해서 검증한다.
2. **엔진이 유일한 계산 출처.** 엑셀·앱이 다른 값을 내면 엔진이 맞다.
3. 근거 없는 값 금지 — 모든 단가·규격은 근거 명시, 추정은 `[추정]` 표기.
4. 실측 벤치마크는 회귀 기준 — 밴드 안에 들어와야 한다.
5. 시세성 값(단가·노임·유가)은 엔진이 조회하지 않고 **인자로 주입**한다.

## 5. 진행 로그 & 다음 작업

### ✅ 완료 — 경로 C (웹 리포트) 1차
엔진 위에 **렌더 전담** 웹 리포트 계층을 얹었다. 구조:
- `render_report.py::compute(inp)` — 엔진 함수만 호출해 4축 결과를 dict로 구조화(계산 로직 없음).
- `render_report.py::render_html(res)` — dict → 자체완결형 HTML. **HTML 안에 JS 계산 없음**(값은 전부 파이썬 엔진 산출).
- `python render_report.py` → 원채원(충남) baseline 리포트, `python render_chuncheon.py` → 춘천(강원) 리포트.

- 원칙 준수: 계산 출처는 엔진 단일, 시세성 값은 `FarmInput`으로 주입.

**경로 C 잔여 개선점**: 현재는 입력을 바꾸려면 `FarmInput`을 코드로 수정해 재생성해야 한다(입력폼 UI 없음). 춘천 케이스의 총공사비·OPEX는 baseline 동일값 `[추정]`으로 둠 — 한랭지 실제 견적이 확보되면 해당 칸만 교체. 춘천 내재해형 설계강도(적설40cm·풍속30m/s)는 통상값 `[확인요망]`, 농사로 '원예특작시설 내재해형 기준'에서 지정값 확인 필요.
