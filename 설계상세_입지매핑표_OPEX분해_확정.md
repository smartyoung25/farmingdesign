# 설계상세 — 입지 지역 매핑표(공공자료) & OPEX 15항목 분해

작성일 2026-07-16 · 상위문서: `작업계획_컨설팅모델_입지설계운영경제성.md`(2026-07-16) · 이 문서는 상위문서 8절의 결정 1·2를 아래와 같이 확정하고, 그 실행설계를 담는다.

## 결정 확정

- **결정 1 (입지 매핑표)**: 공공자료로 확보하는 방향으로 확정. 수동입력은 매핑표에 없는 신규지역에 한해 보조수단으로만 유지.
- **결정 2 (OPEX 분해)**: 이번 라운드에 포함. 단, "항목값 재추정"이 아니라 "기존 lump-sum을 표준 항목체계로 분해"하는 범위로 한정한다(현장에서 바로 쓸 수 있는 범위로 제한 — 근거 없는 세부값을 새로 만들지 않는다).

두 결정 모두 프로젝트 3대 원칙(엔진 단일출처·근거 명시·모래 위 자동화 금지)을 지키는 선에서, 실제 컨설팅 현장에서 즉시 쓸 수 있는 만큼만 설계한다.

## 1. 입지 지역 매핑표 — 공공자료 확보 설계

### 1-1. 실제 확인된 소스 (오늘 조사)

| 우선순위 | 소스 | 성격 | 이번 세션 접근 결과 |
|---|---|---|---|
| **1차** | 「원예·특작시설 내재해 설계기준 및 내재해형 시설규격 등록 등에 관한 규정」(농림축산식품부 행정규칙, 국가법령정보센터) | 온실 전용 지역별 설계하중 근거. `SPEC_TABLE`(32종)의 상위 근거와 동일 계통 | 목록 페이지만 확인, 별표 본문은 이번 세션 WebFetch로 열람 불가(robots 차단·본문 미노출) → **사람이 law.go.kr에서 직접 "별표·서식" 탭 열람 필요** |

| **3차(참고·교차검증 전용)** | 국가건설기준센터 KDS 41 10 05 건축구조기준 — 지역별 기본풍속·설계적설하중 지역계수표 | 일반 건축물 기준(온실 아님). 온실 규정과 수치가 다를 수 있음 | 목록 확인. **직접 적용 금지** — 유리 U값=5.3처럼 "다른 기준을 잘못 가져온" 사고를 반복하지 않기 위한 교차검증 참고자료로만 사용 |
| **4차(대체경로)** | 기상청 기상자료개방포털 — 설계기상자료(재현기간별 최심적설·최대순간풍속) | 매핑표에 없는 신규지역의 대체 추정 근거 | 미조사(필요시 후속) |

### 1-2. 확보 절차 (P0-b, 실행 순서)

1. law.go.kr에서 위 행정규칙의 별표(지역구분표)를 직접 열람 — 자동 크롤링이 막혀 있어 담당자가 한 번은 수기로 열람·저장해야 한다(스크린샷/PDF 저장 권장, 스마트팜스펙 폴더에 축적해온 방식과 동일).
2. 최신 개정 여부 확인 — 오늘 조사에서 "강풍 피해 예방 위한 원예·특작시설 내재해 설계기준 개정"(농식품부 보도자료) 존재를 확인했다. 2014-78호 원문이 이후 개정됐을 가능성이 있으므로 **개정판 우선, 원문과 다르면 개정판으로 SPEC_TABLE도 함께 갱신**해야 한다(신규 이슈 — 상위문서 P0-a 수준의 우선순위로 취급).
3. 농사로 포털에서 기존 케이스 2건(원채원=충남, 춘천=강원)의 지역값을 조회해 별표 값과 대조 — 일치하면 매핑표 신뢰도 확정, 불일치하면 어느 쪽이 최신인지 재확인.
4. 확보한 지역구분표를 `엔진데이터_레지스트리.json`에 `REGION_DESIGN_LOAD` 항목으로 신설 — 출처(고시번호·개정일자)·상태(실측) 태깅.
5. 매핑표에 없는 지역(신규 조성 후보지 등)은 인접 관측소 기상통계로 추정 + `[확인요망]` 태깅 — 자동화선을 넘지 않는다(판단성 취급).

### 1-3. 엔진 모듈 설계

```
def siting_design_load(region_name: str) -> Optional[dict]:
    """행정구역명(시군구) → {snow_cm, wind_ms, source, status}.
    매핑표에 없으면 None — 예외를 던지지 않고 '수동입력 필요'로 상위에서 처리."""
```

- `select_specs(region_snow_cm, region_wind_ms)`는 그대로 둔다(하위호환). `siting_design_load()`는 그 앞단에 붙는 **입력 보조 함수**로만 추가한다 — 계산 로직을 바꾸지 않는다.
- 행정구역 키는 지자체 명칭이 아니라 **법정동코드(표준코드)**로 저장한다. 명칭 매칭은 행정구역 통합·개편에 취약하다(예: 시군 통합).
- 회귀테스트: 매핑표 신설 후 기존 원채원(충남)·춘천(강원) 케이스의 `region_snow_cm`/`region_wind_ms`가 매핑표 조회값과 일치하는지 대조하는 테스트 1건 추가.

### 1-4. 케이스 스키마 v2 (입지 필드)

```json
"site": {
  "region_code": "법정동코드",
  "region_name": "표시용 지명",
  "design_load_source": "고시/개정 근거",
  "design_load_status": "실측 | 확인요망",
  "landuse_zone": "용도지역(육안 확인, 판단성)",
  "grid_connection_note": "[확인요망] 계통연계 여유용량 등"

```
`landuse_zone`(용도지역)·`grid_connection_note`는 **자동 조회하지 않는다.** 토지이음(eum.go.kr)·LURIS는 필지 단위 열람이라 케이스마다 사람이 확인해 채우는 판단성/수동 항목으로 설계한다(2절 신뢰유형 표와 동일 원칙).

## 2. OPEX 15항목 분해 — 이번 라운드 포함 설계

### 2-1. 표준 항목체계의 실제 소스

농촌진흥청 「농산물소득조사」방법론이 표준이다. 오늘 확인한 공공데이터포털 자료 "농촌진흥청_농산물소득분석 조사입력항목코드_20201015"(data.go.kr, ID 15069669)가 원천이며, 대분류는 **직접경비 + 간접경비**로 나뉜다는 것은 확인했으나, **세부 항목명·정확한 개수(15개 여부 포함)는 이번 세션에서 CSV 원문을 직접 받지 못해 확정하지 못했다** — data.go.kr은 목록 페이지만 열람되고 실제 파일 다운로드는 포털 회원가입 후 진행해야 한다.

**이 부분은 추정하지 않는다.** "15항목"이라는 표현은 기존 인수인계서 문구를 그대로 인용한 것일 뿐 이번 조사로 재확인된 숫자가 아니므로, 정확한 항목 리스트 확정을 **P0-d의 1번째 실행항목**으로 명시한다(아래 2-3).

### 2-2. 기존 자산과의 관계 정리 (혼동 방지)

- `스마트팜스펙/` 폴더의 원가계산서(최혁진·이두희·우민재 등)는 **시공비(CAPEX) 항목**(철골공사·피복공사 등)이다. 이번 OPEX(운영비, 매년 반복되는 종묘·비료·농약·에너지·인건비)와는 다른 자료다 — P0 근거문서 흡수 라운드 1·2에서 이미 다룬 자료를 재사용할 수 없다는 뜻이며, **신규로 조사해야 한다.**
- 이 세션에서 확인한 `smartfarm-consulting-report` 스킬은 정확히 이 운영비 영역(농진청 소득조사표 기반 경영성과 분석)을 다루는 기존 자산이다. 그 스킬은 "3대항목 비중"·"kg당 마진" 같은 **비교·진단** 관점이고, 이번에 설계하는 `opex_breakdown()`은 **엔진 계산(finance 연동)** 관점이다 — 역할은 다르지만 **항목체계(무엇을 종묘비/비료비/광열동력비로 볼 것인가)는 반드시 공유**해야 한다. 그렇지 않으면 같은 프로젝트 안에 두 개의 OPEX 기준이 생겨(엑셀 병렬계산기 금지 원칙과 같은 문제) 컨설팅 리포트와 경영성과 보고서의 숫자가 서로 다르게 나올 위험이 있다.

### 2-3. 확보 절차 (P0-d, 실행 순서)

1. 공공데이터포털에서 "농촌진흥청_농산물소득분석 조사입력항목코드" API 신청·CSV 다운로드(사용자 계정으로 진행 필요 — 무료지만 회원가입·활용신청 승인 절차가 있다).
2. CSV의 항목명·코드를 그대로 `엔진데이터_레지스트리.json`에 `OPEX_ITEM_CODES`로 등록(직접경비/간접경비 구분 유지).
3. `smartfarm-consulting-report` 스킬 쪽 자산(`assets/작성템플릿_v2.docx`)에 이미 반영된 항목 표현이 있으면 대조해 용어를 통일한다(스킬 자산이 이 환경에 없으면 다음 세션에서 사용자가 원본 폴더 파일로 대조).
4. 원채원 케이스의 기존 OPEX 총액(186,420,000원)을 항목별로 나눠 채운다 — 이때 항목별 세부값을 새로 추정하지 말고, **확보 가능한 것만 실측/추정 태깅해 채우고 나머지는 "미분류 잔액"으로 남긴다.** 총액이 절대 바뀌면 안 된다(회귀 기준).

### 2-4. 엔진 모듈 설계

```
@dataclass
class OpexBreakdown:
    items: dict[str, float]     # 항목명 → 금액 (직접입력, 시세성)
    unclassified: float          # 총액 - 분류된 항목 합 (근거 부족분, 강제로 0으로 만들지 않음)
    total: float

def opex_breakdown(items: dict[str, float], known_total: float) -> OpexBreakdown:
    """항목별 금액을 합산만 한다(창작 없음). known_total과 항목합이 다르면
    차액을 unclassified로 남겨 '근거 없는 값 금지' 원칙을 지킨다."""
```

- `finance()`는 `opex: float` 파라미터를 그대로 유지하되, `opex_breakdown.total`을 넘겨받는 방식도 허용하도록 얇게 확장한다(하위호환 유지, 기존 회귀 케이스 깨지지 않음).
- **회귀테스트**: (a) 원채원 케이스 항목분해 후 `total == 186,420,000원`(기존 finance 회귀값 불변 확인), (b) `unclassified`가 음수가 되지 않는지(항목합이 총액을 넘지 않는지) 확인하는 가드 테스트.
- 항목별 값 자체(종묘비 얼마 등)는 여전히 **시세성 입력**이며 엔진이 추정하지 않는다 — 결정론 부분은 "합산·비중계산·잔액검증"뿐이다.

### 2-5. 리포트 반영

`build_site.py`의 "운영계획서" 섹션(상위문서 6절)에 항목별 금액·비중(%) 표를 추가한다. `unclassified`가 0보다 크면 리포트에 "근거 미확보 잔액 ○○원(전체의 ○%)"으로 그대로 노출한다 — 감추지 않는 것이 신뢰성 원칙에 부합한다.

## 3. 갱신된 실행 순서 (상위문서 7절 대체)

1. **검증**: `pytest test_engine.py test_registry.py -q` 먼저 실행(습관화).
2. **P0-a**(기존, 최우선 유지): 유리 U값=5.3 단위 의심 해소.
3. **P0-b 1단계**: law.go.kr 행정규칙 별표 직접 열람(사람 개입 필요) + 최신 개정 여부 확인. 개정판 발견 시 SPEC_TABLE도 함께 갱신 검토.
4. **P0-d 1단계**: 공공데이터포털 "농산물소득분석 조사입력항목코드" 계정 신청·CSV 확보.
5. **P0-b·P0-d 후속**: 위 자료 확보되는 대로 레지스트리 등록 → `siting_design_load()`/`opex_breakdown()` 함수 및 회귀테스트 구현(P1).
6. **P1-c**: 케이스 스키마 v2(`site.*`, OPEX 항목 필드) 반영, 원채원 케이스로 시범 채움.
7. **P2**: 4섹션 리포트(입지진단서·설계적정성보고서·운영계획서·경제성분석서)에 신규 필드 반영.

## 4. 남은 리스크 (숨기지 않고 명시)

- law.go.kr·nongsaro.go.kr·k-stat.go.kr는 이번 세션의 자동 조회가 로봇 차단으로 막혀 있다 — **1단계는 사람이 직접 브라우저로 열람해야 진행 가능**하다. 다음 세션에서 이 부분이 비어 있으면 자동화가 실패한 게 아니라 "아직 사람이 안 열어본 것"임을 구분해야 한다.
- 내재해 설계기준이 2014-78호 이후 개정됐을 가능성이 확인됐다 — 매핑표 작업이 곧 SPEC_TABLE 자체의 최신성 재검증으로 이어질 수 있다(작업량 증가 요인, 미리 인지 필요).
- OPEX 표준 항목수("15개")는 아직 미확정 값이다 — CSV 확보 전까지 이 숫자를 리포트나 코드 주석에 확정값처럼 쓰지 않는다.
