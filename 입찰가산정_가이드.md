# 적정 입찰가 산정 가이드 (엔진 도구 활용)

## 0. 목적

공공입찰 준비 Phase 3("공고 후 입찰 준비")에서 가장 실수하기 쉬운 지점은 입찰가를 감(感)으로 정하는 것이다.
너무 낮으면 적격심사에서 "저가심의"(낙찰하한율 미만 응찰 시 감점·소명·실격 대상) 리스크가 생기고,
너무 높으면 경쟁력을 잃는다. 이 문서는 이번 세션에서 만든 `generate_rfq_package()` /
`reconcile_quote()` / `compare_quotes()`를 실제 입찰 준비에 쓰는 절차를 정리한다.

**원칙 재확인**: 엔진은 계산만 한다. "이 가격으로 넣어라"라고 결정해주지 않는다.
`BENCHMARK_BANDS`가 제시하는 건 실측 7건(원채원·우민재·최혁진·이두희 등)에서 나온 "정상 범위"일 뿐,
최종 입찰가 결정은 컨설턴트·시공사의 판단 영역이다.

---

## 1. 왜 이 도구가 유용한가

- `BENCHMARK_BANDS`는 실측 원가 데이터 기반 원/㎡ 정상범위를 갖고 있다 (예: 필름 115,000~240,000원/㎡).
- `benchmark_check()`(`reconcile_quote()`가 내부적으로 호출)가 입찰 예정가를 이 밴드에 대입해
  "정상 / 경계 / 경고(밴드이탈)"를 즉시 판정해준다.
- 밴드 하한에 가깝게 넣으면 가격경쟁력은 높지만 저가심의 리스크가 커지고,
  상한에 가까우면 안전하지만 경쟁에서 밀린다 — 이 트레이드오프를 **숫자로 미리 확인**할 수 있다.

---

## 2. 실전 사용 흐름 (4단계)

### Step 1 — RFQ 패키지 생성 (이 사업의 "기준선" 만들기)

발주 대상 온실의 입지(지역 설계하중)·면적·피복재·형식을 넣어 기준 RFQ를 만든다.
이게 이번 입찰의 "정상 범위가 어디인지"를 보여주는 출발점이다.

```python
import smartfarm_engine as e

rfq = e.generate_rfq_package(
    region_snow_cm=30, region_wind_ms=35,   # siting_design_load()로 지역명→값 조회 가능
    area_m2=2323, cover=e.Cover.FILM, form="연동",
    t_target=10, t_min=-7.8, fr=0.7,
    surface_area_m2=3362,
)
print(rfq.benchmark_band_won_m2)   # 예: (115000, 240000) — 이 사업의 정상 원/㎡ 범위
print(rfq.total_estimate_b_won)    # 골조단독 개산액(방법B) — 참고용 하한 감각
```

지역명만 아는 경우 `siting_design_load("지역명")` → `siting_lookup("지역명", form="연동")`으로
설계하중과 규격 후보까지 한 번에 조회할 수 있다.

### Step 2 — 우리 견적(입찰 예정가) 산정

원가계산서 기준으로 산출한 순공사비/총공사비를 정리한다. 이때 **직접공사비(재+노+경)와
제경비 포함 총공사비를 반드시 분리**해야 한다 — 완전성 검사는 전자, 밴드 판정은 후자를 쓴다.
섞으면 밴드 판정이 왜곡된다(우민재 사례: 직접공사비 456,158,140원 ≠ 총공사비 557,152,000원).

### Step 3 — 밴드 검증

```python
check = e.benchmark_check(
    total_cost=530_000_000,   # 우리가 넣으려는 입찰 예정가(총공사비)
    area_m2=2323,
    cover=e.Cover.FILM,
)
print(check)
# {'unit_won_m2': 228,153, 'band': (115000, 240000), 'status': '정상'}
```

- `정상`: 밴드 안쪽 — 무난.
- `경계`: 밴드 ±10% — 저가심의 대상 여부를 발주기관 적격심사 세부기준(낙찰하한율)으로 별도 확인 필요.
- `경고(밴드이탈)`: 밴드를 크게 벗어남 — 가격을 올리거나(과소 산정 시), 원가 재검토(과다 산정 시).

### Step 4 (선택) — 여러 가격 시나리오 비교

같은 RFQ에 우리 견적의 "보수적 안 / 공격적 안" 등 여러 시나리오를 만들어 나란히 비교할 수 있다.
(주의: `compare_quotes()`는 원래 "여러 업체 견적 비교"용이지만, 시나리오 비교에도 그대로 쓸 수 있다 —
`vendor_name`에 "보수안"/"공격안" 같은 라벨을 넣으면 된다.)

```python
scenarios = [
    e.VendorQuote(vendor_name="보수안(밴드상단80%)", categories={...},
                  direct_cost_total=..., total_with_overhead=520_000_000,
                  area_m2=2323, spec_name=rfq.spec_name),
    e.VendorQuote(vendor_name="공격안(밴드하단110%)", categories={...},
                  direct_cost_total=..., total_with_overhead=485_000_000,
                  area_m2=2323, spec_name=None),
]
comparison = e.compare_quotes(rfq, scenarios)
for row in comparison.rows:
    print(row.vendor_name, row.overall_status, row.unit_won_m2)
```

`compare_quotes()`는 "최저가"·"최고점수"를 정보로만 보여주고 승자를 정하지 않는다 —
최종 입찰가는 사람이 결정한다.

## 3. 체크리스트 (Phase 3 진입 시 바로 쓰는 용도)

- [ ] 발주 대상 지역의 설계하중 확인 (`siting_design_load()` 또는 원문 고시 대조)
- [ ] `generate_rfq_package()`로 기준 RFQ 생성, `benchmark_band_won_m2` 확인
- [ ] 자체 원가계산서에서 직접공사비/총공사비 분리 확정
- [ ] `benchmark_check()`로 입찰 예정가 밴드 판정 — "경고" 나오면 원가 재검토
- [ ] "경계" 판정 시 해당 발주기관의 적격심사 세부기준에서 낙찰하한율(통상 예정가격의 80~87.995% 구간,
      기관·연도별 상이) 확인 — 밴드와 낙찰하한율은 별개 개념이므로 반드시 공고문 원문 확인
- [ ] 필요 시 시나리오 2~3안을 `compare_quotes()`로 나란히 비교해 최종안 선택

---

## 4. 한계 (반드시 인지할 것)

- `BENCHMARK_BANDS`는 표본 7건 기반 — 통계적으로 크지 않다. "정상" 판정이 곧 "이 가격이 낙찰된다"를
  보장하지 않는다. 참고 신호일 뿐이다.
- 낙찰하한율(적격심사 실격 기준)은 **발주기관·연도별 공고문에 명시된 값**을 따라야 한다 —
  엔진의 밴드와는 별개 개념이니 혼동하지 말 것.
- `total_estimate_a_won`(방법A, 온실전체 개산)은 스펙명에 단가가 있을 때만 채워진다 —
  `None`이면 방법B(골조단독)만 참고하고 나머지는 CAPEX 카테고리별로 별도 산정해야 한다.
